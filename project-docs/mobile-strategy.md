# Mobile Strategy

This document captures the mobile strategy for BudgetTracker, including the current state assessment, planned approach, and implementation details.

## Current State

The frontend (`budgettracker.client`) is a React + TypeScript + Vite SPA with Tailwind CSS and Material UI installed. The current mobile readiness:

### What's working
- Viewport meta tag is set correctly (`width=device-width, initial-scale=1.0`)
- Responsive container padding (`sm:px-6`, `lg:px-8`)
- `overflow-x-auto` on tables allows horizontal scrolling as a fallback
- Semantic HTML structure (`<header>`, `<main>`, `<section>`)
- Vector-based Lucide icons (scalable)

### What needs improvement
- **Expenses table**: 6-column table forces horizontal scrolling on small screens. Should convert to a card layout below `md:` breakpoint or hide low-value columns with `hidden md:table-cell`
- **Header text**: Fixed `text-3xl` doesn't scale. Should use `text-xl sm:text-2xl lg:text-3xl`
- **Touch targets**: Interactive elements should be at least 44px tall for comfortable tapping
- **Section padding**: Fixed `px-6` on card interiors should be `px-4 sm:px-6`
- **MUI unused**: `@mui/material` and `@mui/x-charts` are installed but not used. MUI has responsive components (`useMediaQuery`, `Drawer`, `BottomNavigation`) that could improve mobile UX

## Chosen Approach: Capacitor

**Decision**: Use [Capacitor](https://capacitorjs.com/) to wrap the existing React app into native iOS/Android shells for app store distribution. Optionally combine with PWA for web-based install.

### Why Capacitor

Capacitor wraps the existing React + Vite web app in a native WebView for iOS and Android. One codebase produces a website **and** native app store builds.

| Consideration         | Capacitor                                    |
|-----------------------|----------------------------------------------|
| Code reuse            | 100% — no changes to existing React app      |
| App store presence    | Yes (iOS App Store, Google Play)             |
| Native API access     | Yes, via plugins (camera, push, biometrics)  |
| Development effort    | Low — adds to existing project               |
| Performance           | WebView-based (not truly native rendering)   |
| Build requirements    | Xcode for iOS, Android Studio for Android    |

### Project structure after Capacitor setup

```
budgettracker.client/
├── src/                    ← existing React app (unchanged)
├── public/                 ← static assets
├── android/                ← generated by Capacitor
├── ios/                    ← generated by Capacitor
├── capacitor.config.ts     ← points to Vite build output
├── vite.config.ts
└── package.json
```

### Setup steps

```bash
cd budgettracker.client

# Install Capacitor
npm install @capacitor/core
npm install @capacitor/cli -D

# Initialize Capacitor (points to Vite dist/ output)
npx cap init "Budget Tracker" "com.budgettracker.app" --web-dir dist

# Add platforms
npx cap add android
npx cap add ios

# Build the web app, then sync to native projects
npm run build
npx cap sync
```

### Development workflow

```bash
# After making frontend changes:
npm run build          # Build the Vite app
npx cap sync           # Copy web assets to native projects
npx cap open android   # Open in Android Studio
npx cap open ios       # Open in Xcode

# For live reload during development:
npx cap run android --livereload --external
npx cap run ios --livereload --external
```

### API configuration

The native app needs to know where the API lives. In development, Capacitor's live reload can proxy to the local .NET backend. For production, the API base URL should be configurable:

```ts
// budgettracker.client/src/lib/api.ts
// Update the Axios base URL to be environment-aware
const api = axios.create({
    baseURL: import.meta.env.VITE_API_URL || '/api',
});
```

When running as a native app, `VITE_API_URL` would be set to the deployed API URL (e.g., `https://api.budgettracker.com/api`). When running as a web app, it falls back to the relative `/api` path which the Vite proxy (or ASP.NET static files) handles.

## Alternative approaches considered

### PWA (Progressive Web App)
Adds a manifest and service worker so the site is installable from the browser. No app store, no separate codebase. Can be combined with Capacitor.

- **Setup**: `vite-plugin-pwa` with manifest configuration and Workbox service worker
- **Benefit**: Zero additional code, works on any device with a browser
- **Limitation**: No app store presence, restricted native API access (especially iOS push notifications)

**Status**: Not yet implemented. Can be added alongside Capacitor with `vite-plugin-pwa` — the two are complementary.

### React Native / Expo
A completely separate mobile app sharing only the .NET API backend. Requires rewriting the entire UI with React Native components.

- **Rejected**: Too much effort for a project this size. Would require maintaining two frontends.

## Responsive UI improvements (pending)

These frontend changes would improve the experience on both mobile web and Capacitor:

1. **Responsive expenses table** — Convert the 6-column table to cards below `md:` breakpoint, or hide ID/Category ID/Notes columns with `hidden md:table-cell`
2. **Responsive typography** — Scale header from `text-xl` to `text-3xl` across breakpoints
3. **Touch-friendly spacing** — Ensure all interactive elements are at least 44px tall
4. **Tighter mobile padding** — Use `px-4 sm:px-6` on card interiors
